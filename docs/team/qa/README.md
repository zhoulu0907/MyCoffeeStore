## QA 工作流程

### 日常代码审查

```
开发提交 → QA 审查 → 有问题？
                        ↓ 是       ↓ 否
                    反馈问题 → 通过审查
                        ↓
                    开发修复
                        ↓
                    QA 验证
```

### 上线前审查

```
开发完成 → 代码审查 → 测试验证 → 安全检查 → 性能测试 → 上线批准
```

## 问题优先级

| 优先级 | 定义 | 响应时间 | 修复时间 |
|--------|------|----------|----------|
| P0 | 严重问题，影响核心功能或存在安全漏洞 | 立即 | 4小时内 |
| P1 | 重要问题，影响用户体验 | 当天 | 2天内 |
| P2 | 一般问题，不影响核心功能 | 3天内 | 1周内 |
| P3 | 优化建议，提升代码质量 | 1周内 | 下个迭代 |

## 常见问题

### Q1: 什么时候需要代码审查？

所有代码提交都需要经过代码审查，包括：
- 新功能开发
- Bug 修复
- 代码重构
- 配置变更

### Q2: 审查需要多长时间？

- 小规模改动：< 4小时
- 中等规模：< 24小时
- 大规模改动：根据实际情况确定

### Q3: 如何处理审查发现的问题？

1. 查看审查报告中的问题描述
2. 按照优先级修复问题
3. 提交修复后通知 QA 验证
4. 验证通过后合并代码

### Q4: 可以合并未解决 P2/P3 问题的代码吗？

P2 和 P3 问题可以在记录后继续合并，但需要在下个迭代中解决。
